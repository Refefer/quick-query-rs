# Bug 123: Agent Repetition Loop — "Dig Deeper" Drift

## 1. Bug Title and Summary

**Title:** Agent Repetition Loop — "Dig Deeper" Drift

**Summary:** Agents get stuck in iterative loops where they re-perform the same investigation steps (e.g., querying logs, checking databases) with slightly different wording across tool calls instead of converging on answers. Symptoms include repeated self-referential statements like *"I'm digging deeper..."* and *"Let me dig deeper into..."* appearing multiple times with near-duplicate tool call arguments.

---

## 2. Symptoms

The following symptoms have been observed in production and test environments:

- **Repetitive thought patterns** — Assistant responses repeat similar thoughts with slight variations, e.g.:
  ```
  query("find user 123")
  → later: query("search for customer 123")
  → even later: query("look up user record 123")
  ```

- **Near-duplicate tool calls** with minor wording changes instead of new investigation directions. Each call is technically distinct but semantically equivalent to a prior one.

- **Self-referential statements about "digging deeper"** appear repeatedly in the conversation history:
  - *"I'm digging deeper into this..."*
  - *"Let me dig deeper into the logs..."*
  - *"I need to dig even deeper..."*

- **No progress toward the original question** despite many iterations. The agent circles the same data sources with different phrasing rather than synthesizing findings or pivoting strategy.

- **Compaction passes fail to deduplicate.** Each compaction cycle re-emits identical or near-identical findings, so the compressed context carries forward all the noise instead of distilling it.

---

## 3. Root Cause Analysis

Four interacting issues combine to cause this bug. The first two are primary drivers; the latter two are contributing factors that exacerbate the core problem.

### Issue 1: Observation Log Double-Injection (HIGH severity)

**Location:** `crates/qq-core/src/agent.rs`, lines ~983–997

The observation log is append-only and injected into the system prompt every iteration with only the label `"## Observation Log (prior context)\n\n{obs_log}"`. There is zero guidance about what this log represents or how to treat it — "(prior context)" is meaningless to an LLM. Stale observations from dozens of iterations ago are re-sent every turn, competing with fresh context. The LLM sees old findings but has no instruction to avoid repeating work already done, so it interprets them as *"context I should act on"* rather than *"context that tells me what's already been investigated."*

### Issue 2: Observer Blind Spot (HIGH severity)

**Location:** `crates/qq-cli/src/compaction.rs`, lines ~10–53 (`OBSERVER_PROMPT`)

The observer only receives the drained slice of messages; it never sees the accumulated `observation_log`. It cannot deduplicate — it re-emits identical findings every compaction pass. Each time the conversation is compacted, the observer summarizes what happened but has no memory of what was already captured in prior passes, so it produces overlapping summaries that add noise rather than clarity.

### Issue 3: `find_safe_split_point()` Partial Overlap (MEDIUM severity)

**Location:** `crates/qq-core/src/observation.rs`

Compaction backs up past tool-call sequences to avoid breaking atomic groups, leaving some messages raw while neighbors are compressed — creating interleaved compressed + raw content covering overlapping ground. This partial overlap means the LLM sees both a compressed summary *and* raw messages that cover similar information, creating confusion about what has already been investigated and reinforcing the repetition loop.

### Issue 4: No Cross-Iteration Tool Call Deduplication (MEDIUM severity)

**Location:** `crates/qq-core/src/agent.rs`, `RepetitionDetector` struct (lines ~80+)

The `RepetitionDetector` only tracks exact `(tool_name, canonical_args)` hashes per iteration; it resets each turn. Canonicalization just strips whitespace — `"find user 123"` ≠ `"search for customer 123"`. The detector has no concept of semantic similarity or even tool-type usage patterns across iterations. It can only catch byte-identical repeated calls within a single agent turn, which is insufficient for drift detection.

## 5. Config Defaults

Configuration defaults from `crates/qq-core/src/observation.rs`:

| Context Window | message_threshold_bytes | observation_threshold_bytes | preserve_recent |
|----------------|------------------------|----------------------------|-----------------|
| 128K (default) | ~7,680 bytes | 32,000 bytes | 6 |
| 200K | ~8,000 bytes | 24,000 bytes | 6 |
| 64K (small) | 30,000 bytes | 100,000 bytes | 6 |

> **Note:** There is an asymmetry in these defaults. Smaller context windows have *higher* thresholds (30 KB / 100 KB) while larger contexts have *lower* ones (~7.6 KB / 32 KB). This means compaction triggers more aggressively on smaller windows where there is less room to spare — potentially amplifying the effects of Issues #2 and #3 (observer blind spot and partial overlap) precisely when the system has the least capacity to absorb redundant information.

## 6. Proposed Fixes (Ranked by Effort/Impact)

### Fix 1: Clarifying Instructions at Injection Time

**Priority:** Quick Win · **Effort:** LOW · **Impact:** HIGH

**What:** Change the system prompt injection for the observation log to explain what it is and how to treat it. Add explicit instruction to avoid repeating work already captured.

**Where:** `crates/qq-core/src/agent.rs`, lines ~982–997

**Implementation:** Replace the bare injection with a structured header that includes behavioral guidance:

```diff
- "## Observation Log (prior context)\n\n{obs_log}"
+ "## Prior Observations\n\n"
+ + "These are observations captured from completed investigation steps.\n"
+ + "Do not repeat investigations already covered in this log — focus on new findings.\n\n"
+ + "{obs_log}"
```

**Risk:** Very low — text-only change to the prompt template. Should be validated with existing agent tests.

---

### Fix 2: Observer Awareness

**Priority:** Medium · **Effort:** MEDIUM · **Impact:** HIGH

**What:** Pass the accumulated `observation_log` to the observer so it can deduplicate against existing entries. Update the `OBSERVER_PROMPT` with rules about checking for already-captured findings.

**Where:** `crates/qq-cli/src/compaction.rs` — `OBSERVER_PROMPT` template (lines ~10–53) and compaction logic

**Implementation:**
- Modify the compaction function signature or context struct to include `prior_observation_log`.
- Update `OBSERVER_PROMPT` with an instruction such as: *"Before adding a new finding, check if it duplicates an existing observation in the prior observation log. If so, skip it."*
- Consider passing the current `observation_log` as part of the compaction input alongside the drained messages.

**Risk:** Low-medium. Changes compaction logic; must ensure the observer still captures all unique findings. Test edge cases where observations are borderline duplicates.

---

### Fix 3: Observation Log Bounded Growth

**Priority:** Lower · **Effort:** LOW–MEDIUM · **Impact:** MEDIUM

**What:** Cap the append-only observation log — implement rotation or truncation when it exceeds a size limit. Alternatively, use the reflection pass more aggressively to compress rather than just accumulate.

**Where:** `crates/qq-core/src/observation.rs`

**Implementation options:**

| Option | Approach | Complexity |
|--------|----------|------------|
| A | Sliding window on the observation_log that drops entries older than N iterations | Low |
| B | When the log exceeds `observation_threshold_bytes`, run a compression pass focused only on the observation log | Medium |
| C | Add a "reflection" step after M consecutive non-productive iterations that summarizes and deduplicates the entire log | Medium–High |

**Risk:** Medium. Truncation could lose important context if not done carefully. The sliding window approach (Option A) is safer than hard truncation. Options B/C provide more control but introduce additional complexity.

---

### Fix 4: Cross-Iteration Tool Call Pattern Detection

**Priority:** High · **Effort:** MEDIUM–HIGH · **Impact:** HIGH

**What:** Track tool-type usage across iterations rather than only exact matches within a single iteration. Flag when any single tool has been called N+ times with different arguments. Add a soft-warning before a hard-block.

**Where:** `crates/qq-core/src/agent.rs` — `RepetitionDetector` struct (lines ~80+)

**Implementation:**
- Extend `RepetitionDetector` to maintain cross-turn history of `(tool_name, arg_summary)` pairs, where `arg_summary` uses fuzzy matching or semantic hashing.
- Track tool-type frequency across the full session (not just per-turn).
- When a tool is called N+ times (configurable; default 3–5) with semantically different args for the same purpose, emit a soft-warning to the LLM before hard-blocking. Example warning: *"You've already attempted X different approaches using `[tool_name]`; consider a new strategy or synthesize what you know."*
- Consider adding a "tool fatigue" signal as part of the injected context.

**Risk:** Medium-high. Fuzzy matching adds complexity; false positives/negatives are possible. Start with exact tool-name + arg-length heuristics before adding semantic analysis. Should be gated behind a config flag initially.

2. **Compaction overlap test** — Run compaction over a long conversation history and verify that successive passes produce non-overlapping summaries.

3. **Measurement tests** — Measure conversation length (token count) and iteration count before/after applying fixes to quantify improvement.

### Regression Tests

1. Add existing bug-triggering scenarios from user reports as regression test cases.
2. Verify compaction produces non-overlapping summaries across multiple passes.
3. Ensure observation log injection includes guidance text in all code paths.

## 9. Implementation Order Recommendation

The recommended order balances speed of delivery against root-cause impact:

1. **Fix 1** — Quick prompt injection change. Fastest to implement, immediate impact on LLM behavior. Low risk makes it ideal as a first deployment.
2. **Fix 4** — Cross-iteration tool call pattern detection. Addresses the root cause directly by catching drift at its source. Can be implemented incrementally (heuristics → fuzzy matching).
3. **Fix 2** — Observer awareness. Requires changes to the compaction interface but eliminates duplicate findings in compressed context.
4. **Fix 3** — Observation log bounded growth. Lowest priority since Fixes 1 and 4 should naturally reduce observation log growth by preventing the repetition loop from starting in the first place.
